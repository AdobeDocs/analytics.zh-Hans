# Adobe Analytics 中的低流量值

当一个报表有许多唯一值时，Adobe 提供了功能来确保在报表中显示最重要的值。在大约500,000个现有值之后收集的唯一变量值列在标记为的维度项目下 **[!UICONTROL 低流量]**.

## [!UICONTROL 低流量]的工作方式

* 如果变量在给定月内未达到 500,000 个唯一值，则报告不受影响。
* 当变量达到500,000个唯一值时，数据将开始在 [!UICONTROL 低流量]. 超出此阈值的每个值都遵循以下逻辑：
   * 如果某个值已在报表中，则正常添加到该值。
   * 如果某个值尚未在报表中显示，则该值最初会在 [!UICONTROL 低流量] 维度项目。
   * 如果某个值在 [!UICONTROL 低流量] 会在该月的某个位置以两位数显示，它开始被识别为自己的维度项目。 在达到阈值之前收集的实例仍低于 [!UICONTROL 低流量]. 确切阈值具有许多依赖关系，例如处理报表包数据的服务器数量以及每个维度项目实例之间的时间。
* 如果报表包的唯一值超过1,000,000个，则会应用更激进的过滤。 唯一值需要以三位数表示的实例，才能被识别为其自己的维度项目。

此逻辑允许Adobe优化报表功能，同时仍允许您的组织报告当月晚些时候收集的关键维度项目。 例如，您的组织运行的网站上包含数百万篇文章，并且一篇新文章在月末很受欢迎（超过两个唯一阈值后）。 您仍然可以分析该文章的性能，而不将其存储在 [!UICONTROL 低流量]. 请注意，此逻辑并非用于取消存储每天或每月获得一定数量页面查看的所有内容。

>[!NOTE]
>的 [页面](../components/dimensions/page.md) 维度使用多个后端列，所有后端列都计入唯一阈值，包括 `pagename`, `page_url`, `first_hit_pagename`, `first_hit_page_url`, `visit_pagename`, `visit_page_url`和 `click_context`. 这些后端列可能会导致 [!UICONTROL 低流量] 逻辑，以便在工作区中的独特页面维度项目数量达到500,000之前应用。

## 更改唯一限制阈值

默认情况下，阈值限制为 500000 和 1000000 个唯一值。这些限制可以按变量更改。请联系Adobe客户关怀或贵组织的客户经理以请求进行此更改。 请求更改时，应包括：

* 报表包 ID
* 要增加阈值的变量
* 所需的第一和第二阈值

更改阈值可能会影响报告性能。Adobe 强烈建议在请求增加变量中的唯一值时应做出准确的判断。仅增加对贵组织的报告需求至关重要的变量的唯一限制。

低流量阈值在 Analytics UI 中不可见。如果您希望了解有关现有阈值的更多信息，请联系Adobe客户关怀团队。

## 使用低流量的组件和其他功能

不同的功能处理低流量值的方式不同。

* **Data Warehouse：** Data Warehouse 报表中的唯一值数量没有限制。其独特的架构允许报告任意数量的唯一值。
   * 在某些有限情况下，仍功能出现低流量值。示例包括列表变量、列表属性、推销 eVar 和营销渠道详细信息维度。
* **分段：**&#x200B;如果区段标准包含具有大量唯一值的变量，则将不包含低流量下捕获的值。
* **分类：**&#x200B;分类报表也受唯一限制的约束。如果分类的父变量值包含在低流量下，则该值不会进行分类。
   * 通过导入器获取的低流量分类值可以在 Data Warehouse 中查看。<!-- AN-115871 -->
   * 通过规则生成器获取的低流量分类值&#x200B;*不能*&#x200B;在 Data Warehouse 中查看。<!-- AN-122872 -->
